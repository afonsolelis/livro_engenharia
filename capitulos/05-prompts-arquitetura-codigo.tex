\chapter{Prompts para Arquitetura e Código}

\section{Introdução}

Prompt engineering não é ``pedir com jeitinho'' para a IA. É uma disciplina de
engenharia. Assim como um engenheiro civil não pede ao calculista para
``fazer uma fundação boa'', o engenheiro de software não deveria pedir ao LLM
para ``escrever um código bom''. Especificidade é poder.

Um prompt bem escrito funciona como uma especificação formal do que você
precisa. Essa afirmação merece reflexão. Quando um engenheiro de software redige
um documento de requisitos, ele sabe que cada ambiguidade será interpretada de
forma diferente por cada desenvolvedor, e que o custo de corrigir um
mal-entendido cresce exponencialmente à medida que o software avança no pipeline
de desenvolvimento. Com LLMs, a dinâmica é idêntica: cada ambiguidade no prompt
é uma bifurcação onde o modelo pode seguir por um caminho que você não pretendia.
A diferença é que, enquanto um desenvolvedor humano pode pedir esclarecimentos,
o LLM simplesmente escolhe a interpretação mais provável dado seu treinamento.
Essa interpretação pode ser radicalmente diferente do que você tinha em
mente. Tratar o prompt como especificação significa aplicar ao texto as mesmas
virtudes que aplicamos a requisitos de software: precisão, completude,
ausência de contradições e testabilidade.

Assim como escrever testes ou documentar APIs, prompt engineering é uma
habilidade que melhora com prática deliberada e reflexão sobre resultados. Não se
trata de talento inato para ``conversar com a máquina''. Trata-se de um
conjunto de técnicas concretas que podem ser aprendidas, praticadas e refinadas.
O engenheiro que dedica tempo a estudar como LLMs processam texto, quais
estruturas de prompt produzem respostas mais consistentes e como iterar sobre
resultados insatisfatórios desenvolve uma vantagem cumulativa. Cada prompt
bem-sucedido é um template reutilizável; cada falha é uma lição sobre os limites
e as idiossincrasias do modelo.

O retorno sobre investimento dessa habilidade é concreto e mensurável. Cinco
minutos adicionais investidos na redação de um prompt bem estruturado economizam
trinta minutos de ida e volta com a IA e revisão de código inadequado. Esse
cálculo não é hiperbólico: quando um prompt vago gera código que ``quase''
resolve o problema, o engenheiro gasta tempo lendo código que não era o que
queria, identificando o que está errado, formulando um novo prompt, esperando a
resposta, lendo novamente, e assim por diante. Quando o prompt é preciso desde o
início, o primeiro resultado já está próximo do desejado, e o ciclo de refinamento
é curto e produtivo.

Vale lembrar que, no Capítulo 4, estabelecemos que a IA é uma parceira de
engenharia. Neste capítulo, aprenderemos a \textbf{comunicar} com essa parceira
de forma eficaz. A conexão é direta: de nada adianta ter o melhor copiloto do
mundo se você não sabe articular para onde quer voar. A comunicação com LLMs
segue os mesmos princípios da comunicação entre humanos em contextos de
engenharia (clareza, contexto, especificidade), com a diferença crucial de
que o LLM não possui intuição para preencher lacunas. Tudo o que ele não recebe
explicitamente, ele inventa a partir de padrões estatísticos.

A diferença entre um engenheiro que usa IA de forma produtiva e um que perde
tempo com ela está, quase sempre, na qualidade dos prompts. Essa é a tese central
deste capítulo, e cada seção que se segue oferece ferramentas concretas para
construir essa qualidade.

\section{Anatomia de um Bom Prompt}
% ============================================================

Um prompt eficaz não é um parágrafo solto. É uma estrutura com componentes
identificáveis, cada um com uma função específica. Assim como um método bem
projetado tem parâmetros nomeados com tipos claros e um contrato documentado, um
prompt eficaz possui seções identificáveis que, juntas, eliminam a ambiguidade e
direcionam o modelo para o resultado desejado.

A analogia com programação é mais profunda do que parece. Quando escrevemos uma
função, definimos seus parâmetros de entrada, seu tipo de retorno, suas
pré-condições e pós-condições. Quando escrevemos um prompt, fazemos a
mesma coisa: definimos o contexto de entrada, o formato de saída esperado, as
restrições que devem ser respeitadas e o critério de qualidade. A diferença é que,
em programação, o compilador nos força a ser explícitos; em prompting, a
responsabilidade é inteiramente nossa.

Cada elemento dessa tabela merece atenção individual. O \textbf{contexto} é o
alicerce sobre o qual todo o resto se constrói. Sem ele, o modelo opera no vácuo,
produzindo respostas genéricas que poderiam se aplicar a qualquer sistema. Com
ele, o modelo pode particularizar suas respostas, considerando as restrições reais
do seu cenário. O \textbf{papel} é um mecanismo poderoso porque ativa, no modelo,
conjuntos diferentes de conhecimento e estilo. Um ``arquiteto sênior'' tende a
produzir respostas mais cautelosas, com mais trade-offs explícitos; um
``desenvolvedor pragmático'' tende a produzir respostas mais diretas e
implementáveis. A \textbf{instrução} deve usar verbos no imperativo e ser
inequívoca: ``analise'', ``compare'', ``implemente'', ``refatore''. As
\textbf{especificidades} são as restrições que transformam uma solução genérica em
uma solução viável para o seu contexto. O \textbf{formato de saída} elimina a
necessidade de reformatar a resposta e, mais importante, força o modelo a
organizar seu raciocínio de uma forma que você pode avaliar. Finalmente, os
\textbf{exemplos} são o mecanismo mais poderoso de todos, porque mostram ao
modelo o que você espera. Modelos de linguagem são, acima de tudo,
máquinas de reconhecimento e replicação de padrões.

\subsection{O Framework CRISP}

Para facilitar a memorização e a aplicação sistemática dos elementos de um bom
prompt, propomos o framework \textbf{CRISP}. O acrônimo não é apenas um recurso
mnemônico: cada letra representa um pilar cuja ausência compromete
significativamente a qualidade do resultado. Vejamos cada um em profundidade.

A letra \textbf{C} representa \textbf{Context}, o cenário, o sistema e o domínio
de negócio em que o problema está inserido. Sem contexto, a IA generaliza; com
contexto, ela particulariza. Considere a diferença entre perguntar ``como
implementar autenticação?'' e ``como implementar autenticação em um sistema de
saúde que precisa atender à LGPD, onde os dados de pacientes são classificados
como sensíveis e o sistema roda em infraestrutura on-premises por exigência
regulatória?''. A segunda pergunta, rica em contexto, produz uma resposta que
considera criptografia em repouso, audit trails, tokens com escopo limitado e
conformidade regulatória. A primeira pergunta jamais evocaria esses elementos.
O contexto funciona como uma lente que foca a atenção do modelo. Internamente,
os LLMs funcionam por atenção: cada token gerado é influenciado por todos os
tokens anteriores, com pesos que variam de acordo com a relevância percebida.
Quando você fornece contexto rico, está essencialmente aumentando o peso dos
tokens relevantes e diminuindo o peso dos irrelevantes. O modelo ``presta mais
atenção'' nas partes do seu treinamento que são relevantes para o seu cenário.
Um contra-exemplo ilustrativo: pedir ``implemente um cache'' sem contexto pode
gerar um cache em memória com HashMap. Isso seria correto em um exercício acadêmico, mas
desastroso em um sistema distribuído com múltiplas instâncias. O contexto
``sistema com 8 instâncias atrás de um load balancer'' imediatamente direciona o
modelo para Redis, Memcached ou outra solução distribuída.

A letra \textbf{R} representa \textbf{Role}, a persona que a IA deve assumir. Esse
é um dos mecanismos mais subestimados do prompt engineering. Quando você diz ``atue
como um arquiteto sênior'', não está apenas pedindo formalidade; está ativando
um conjunto de heurísticas no modelo. Um ``arquiteto sênior'' tende a considerar
escalabilidade, manutenibilidade e trade-offs; um ``desenvolvedor júnior
explicando para outro júnior'' tende a simplificar conceitos e usar analogias. A
escolha do papel depende do que você precisa: se quer explorar alternativas
complexas, peça um arquiteto sênior; se quer validar se uma explicação é clara o
suficiente para onboarding, peça a perspectiva de um desenvolvedor iniciante.
O papel também pode ser usado para simular personas de stakeholders: ``atue como o
CTO que precisa justificar o investimento técnico para o board'' produz uma análise
com foco em ROI e risco de negócio, enquanto ``atue como o tech lead que precisa
convencer a equipe'' produz argumentos técnicos e pragmáticos. O contra-exemplo
aqui é não definir papel nenhum. Sem papel, o modelo assume uma persona ``neutra''
que tenta ser genericamente útil, o que frequentemente resulta em respostas
superficiais que não se comprometem com nenhuma perspectiva.

A letra \textbf{I} representa \textbf{Instructions}, o que exatamente você quer
que a IA faça. Aqui, a precisão do verbo importa. ``Analise'' é diferente de
``compare'', que é diferente de ``implemente'', que é diferente de ``refatore''.
Cada verbo implica uma operação cognitiva distinta: analisar é decompor em partes;
comparar é colocar lado a lado e avaliar critérios; implementar é produzir código
funcional; refatorar é mudar a estrutura preservando o comportamento. Verbos no
imperativo eliminam ambiguidade sobre a ação esperada. Um erro comum é usar verbos
vagos como ``trabalhe com'' ou ``lide com''. Esses verbos não comunicam
nenhuma ação específica e deixam o modelo livre para interpretar de qualquer
forma. Outro erro é empilhar múltiplas instruções conflitantes: ``analise e
implemente e documente e teste'' em uma única sentença. Quando há múltiplas ações,
é melhor numerá-las explicitamente ou usar prompt chaining para separar cada uma
em seu próprio momento.

A letra \textbf{S} representa \textbf{Specifics}, as restrições técnicas,
requisitos não funcionais, linguagem, framework e padrões obrigatórios. Se
Context é o ``onde'' e Instructions é o ``o quê'', Specifics é o ``dentro de
quais limites''. Essa seção é onde o engenheiro de software se diferencia do
usuário casual de IA. Um usuário casual pede ``crie uma API''; um engenheiro
especifica ``API REST em Python 3.12, FastAPI, com autenticação OAuth2, rate
limiting de 100 req/min por usuário, respostas em JSON:API spec, e latência P99
abaixo de 200ms''. Cada restrição adicionada elimina um grau de liberdade do
modelo, direcionando-o para soluções que são não apenas corretas em abstrato, mas
viáveis no seu contexto concreto. Restrições de negócio são tão importantes
quanto restrições técnicas: ``o cálculo de frete deve considerar a tabela de
preços dos Correios vigente em 2024'' é uma restrição que impede o modelo de
inventar regras de cálculo fictícias. O contra-exemplo é a ausência total de
restrições, que resulta em código que pode estar tecnicamente correto mas é
inutilizável no seu projeto: código em JavaScript quando você usa TypeScript,
código sem tratamento de erro quando sua política exige fail-fast, código com
dependências externas quando seu projeto precisa ser air-gapped.

A letra \textbf{P} representa \textbf{Parameters}, o formato da resposta, o nível
de detalhe, o tom, o comprimento e a inclusão de exemplos ou diagramas. Esse
elemento controla a \textit{forma} da resposta, enquanto os anteriores controlam
o \textit{conteúdo}. Definir parâmetros de saída é como definir o tipo de retorno
de uma função: sem ele, você não sabe o que esperar. ``Responda em formato ADR''
produz uma resposta estruturada e comparável; ``responda como quiser'' produz
texto livre que pode ser um parágrafo ou uma dissertação de doutorado. Parâmetros
úteis incluem: formato (tabela, lista, diagrama Mermaid, ADR, RFC), nível de
detalhe (resumo executivo vs. especificação técnica completa), comprimento
(``em no máximo 500 palavras'' ou ``com no máximo 3 páginas''), e inclusão de
artefatos (``inclua diagrama de sequência'' ou ``inclua exemplos de código''). Um
parâmetro frequentemente negligenciado é o tom: ``explique como se estivesse em
uma reunião de design review'' produz um tom diferente de ``explique como se
estivesse escrevendo documentação oficial''. O primeiro é mais informal e
opinativo; o segundo é mais formal e objetivo.

\begin{lstlisting}[language={}, caption={Exemplo de prompt usando o framework CRISP}]
[Context]
Estou desenvolvendo um sistema de pagamentos para uma fintech brasileira.
O sistema processa 10.000 transacoes/dia via Pix e cartao de credito.
Usamos Java 21, Spring Boot 3, PostgreSQL 16 e AWS.

[Role]
Atue como um arquiteto de software senior com experiencia em sistemas
financeiros e compliance do Banco Central do Brasil.

[Instructions]
Proponha uma arquitetura para o modulo de conciliacao financeira que
roda diariamente, comparando transacoes internas com extratos bancarios.

[Specifics]
- Deve suportar idempotencia (reprocessamento seguro)
- Tempo maximo de execucao: 30 minutos para 10k transacoes
- Deve gerar relatorio de divergencias
- Precisa de audit trail completo
- Integracao com S3 para receber extratos em CSV

[Parameters]
Responda com:
1. Diagrama de componentes (em formato Mermaid)
2. Descricao de cada componente (2-3 frases)
3. Principais trade-offs da abordagem escolhida
4. Riscos e mitigacoes
\end{lstlisting}

Observe como esse prompt exemplifica cada letra do CRISP em ação. O contexto
estabelece o domínio (fintech brasileira), a escala (10 mil transações/dia) e a
stack tecnológica. O papel ativa conhecimento sobre sistemas financeiros e
regulação do Banco Central. A instrução é precisa: ``proponha uma arquitetura para
o módulo de conciliação financeira''. As especificidades definem restrições
concretas e mensuráveis. E os parâmetros determinam exatamente o formato da
resposta esperada. Não há espaço para o modelo produzir uma resposta genérica
sobre ``como fazer conciliação financeira''; ele é forçado a produzir uma
arquitetura específica, com componentes identificáveis, trade-offs explícitos e
riscos mapeados. Esse é o poder do CRISP: transformar a vagueza em especificação.

\subsection{O Poder dos Exemplos (Few-Shot Prompting)}

Exemplos valem mais que descrições. Quando você mostra à IA \textbf{como} quer
a resposta, ela replica o padrão com mais fidelidade do que quando você apenas
\textbf{descreve} o que quer. Essa é uma consequência direta de como os LLMs
funcionam internamente.

Para entender por que exemplos são tão poderosos, é preciso compreender o
mecanismo fundamental dos modelos de linguagem. LLMs são modelos de
predição de sequências: dado um contexto (uma sequência de tokens), eles preveem
o próximo token mais provável. Quando você fornece um exemplo no prompt, está
adicionando ao contexto um padrão completo de entrada-e-saída. O modelo, ao gerar
sua resposta, ``vê'' esse padrão e o utiliza como template implícito. É como
fornecer um caso de teste ao programador antes de pedir a implementação. O
padrão esperado fica inequívoco.

A técnica \textbf{zero-shot} envolve não fornecer nenhum exemplo, dependendo
inteiramente da compreensão da instrução textual pelo modelo. Funciona bem para
tarefas simples e bem definidas (``converta esta temperatura de Celsius para
Fahrenheit''), mas falha quando o formato de saída é específico ou quando há
convenções implícitas que o modelo não pode adivinhar. A técnica \textbf{one-shot}
adiciona um único exemplo, e o salto de qualidade é notável: mesmo um único
exemplo já comunica formato, tom, nível de detalhe e convenções de nomenclatura.
A técnica \textbf{few-shot}, com dois ou três exemplos, é ideal para tarefas com
padrão específico de saída, como gerar ADRs, documentação padronizada ou logs
estruturados. Três exemplos são geralmente suficientes para estabelecer um
padrão robusto; mais do que cinco exemplos raramente traz benefício adicional e
consome tokens preciosos da janela de contexto.

A regra prática é simples: se o formato importa, inclua pelo menos um exemplo
completo. O custo em tokens compensa amplamente pela economia em iterações. Um
prompt com exemplo que produz o resultado correto na primeira tentativa é sempre
mais eficiente do que um prompt sem exemplo que exige três ou quatro rodadas de
refinamento. Pense nos exemplos como testes de aceitação do seu prompt: eles
demonstram o resultado esperado de forma inequívoca.

\section{Prompts para Design Arquitetural}
% ============================================================

Arquitetura é onde a IA pode ser mais útil e, ao mesmo tempo, mais perigosa. Útil porque
explora alternativas rapidamente, considerando combinações que um humano
levaria horas para analisar. Perigosa porque pode gerar respostas
superficialmente convincentes mas fundamentalmente inadequadas.

O perigo reside em um fenômeno que os pesquisadores chamam de ``fluência sem
fundamentação''. LLMs são extraordinariamente bons em produzir texto que \textit
{parece} competente: usa a terminologia correta, segue a estrutura lógica esperada,
cita trade-offs plausíveis. Mas essa fluência pode mascarar raciocínios
superficiais. O modelo pode recomendar Event Sourcing porque ``parece'' adequado
para sistemas financeiros, sem realmente considerar se sua equipe de três
desenvolvedores juniores tem condição de operar essa arquitetura. É por isso que
prompts para design arquitetural precisam ser especialmente ricos em contexto e
restrições: quanto mais informação o modelo tem sobre seu cenário real, mais
difícil é para ele produzir recomendações genéricas que ignoram suas restrições.

A estratégia fundamental para usar IA em decisões arquiteturais é tratá-la como
uma ferramenta de \textbf{exploração}, não de \textbf{decisão}. A IA gera
alternativas, analisa trade-offs e identifica riscos. O engenheiro humano toma a
decisão final, usando sua compreensão do contexto organizacional, da dinâmica da
equipe e das prioridades de negócio. Nenhum prompt, por mais rico
que seja, consegue capturar esses fatores completamente.

\subsection{Template: Explorando Alternativas Arquiteturais}

O prompt a seguir foi projetado para explorar o espaço de soluções de forma
sistemática. Cada seção tem uma razão de ser. O bloco de contexto fornece as
informações que o modelo precisa para particularizar suas recomendações. Os
requisitos funcionais e não funcionais definem o ``o quê'' e o ``quão bem''. As
restrições eliminam soluções teoricamente elegantes mas praticamente inviáveis.
O formato de saída, ao solicitar pelo menos três alternativas com prós,
contras, complexidade, risco e nota de adequação, força o modelo a fazer
análise comparativa em vez de simplesmente recomendar sua ``resposta favorita''.

\begin{lstlisting}[language={}, caption={Prompt para exploração de alternativas arquiteturais}]
Contexto: [descreva o sistema, dominio, escala, equipe]

Preciso decidir a arquitetura para [componente/modulo/sistema].

Requisitos funcionais:
- [RF1]
- [RF2]
- [RF3]

Requisitos nao funcionais:
- [RNF1: ex. latencia < 100ms p99]
- [RNF2: ex. disponibilidade 99.9%]
- [RNF3: ex. equipe de 4 devs]

Restricoes:
- [Stack atual: linguagem, cloud, banco]
- [Orcamento: X/mes]
- [Prazo: Y semanas]

Para cada alternativa arquitetural viavel:
1. Descreva a abordagem em 3-5 frases
2. Liste pros e contras (minimo 3 de cada)
3. Estime complexidade de implementacao (baixa/media/alta)
4. Identifique o principal risco
5. De uma nota de adequacao de 1-10 para este cenario especifico

Apresente pelo menos 3 alternativas.
\end{lstlisting}

O que torna esse template eficaz é a combinação de abertura exploratória com
estrutura analítica. Ao pedir ``pelo menos 3 alternativas'', evitamos o viés de
ancoragem, a tendência do modelo de se fixar na primeira solução que vem à
mente. Ao exigir prós e contras para cada alternativa, forçamos uma análise
equilibrada. A nota de adequação ``para ESTE cenário específico'' é crucial:
sem ela, o modelo pode dar nota alta para uma solução que é excelente em
abstrato mas inadequada para sua equipe e suas restrições.

\subsection{Template: Análise de Trade-offs}

Decisões arquiteturais são, fundamentalmente, decisões sobre trade-offs. Não
existe arquitetura perfeita; existe a arquitetura que faz os trade-offs certos
para o seu contexto. Este template é desenhado para quando você já identificou as
alternativas e precisa de uma análise comparativa rigorosa. A definição de eixos
de avaliação (performance, escalabilidade, complexidade operacional, curva de
aprendizado, custo e flexibilidade) garante que a comparação é feita em dimensões
relevantes e consistentes.

O bloco sobre ``momento do produto'' é particularmente importante. A arquitetura
ideal para um MVP é radicalmente diferente da arquitetura ideal para um sistema em
escala. Informar ao modelo se você está em fase de validação, crescimento ou
manutenção muda fundamentalmente as recomendações.

\begin{lstlisting}[language={}, caption={Prompt para análise de trade-offs arquiteturais}]
Atue como arquiteto senior revisando uma decisao tecnica.

Estamos decidindo entre:
- Opcao A: [descreva]
- Opcao B: [descreva]
- Opcao C: [descreva] (opcional)

Contexto do sistema: [descreva]
Equipe: [tamanho, senioridade, experiencia]
Momento do produto: [MVP, escala, manutencao]

Analise cada opcao nos seguintes eixos:
1. Performance (latencia, throughput)
2. Escalabilidade (horizontal, vertical)
3. Complexidade operacional (deploy, monitoramento)
4. Curva de aprendizado para a equipe
5. Custo (infraestrutura + desenvolvimento)
6. Flexibilidade para mudancas futuras

Para cada eixo, classifique como: Excelente / Bom / Aceitavel / Ruim.
Ao final, faca uma recomendacao fundamentada para ESTE cenario especifico.
\end{lstlisting}

A eficácia desse prompt reside na classificação categórica (Excelente, Bom,
Aceitável, Ruim) e na exigência de recomendação fundamentada para o cenário
específico. A classificação categórica torna a comparação visual e imediata:
basta olhar para a tabela gerada e identificar onde cada opção brilha e onde
falha. A fundamentação específica impede que o modelo recorra ao clichê ``depende
do contexto'' sem se comprometer.

\subsection{Template: Geração de ADRs}

Architecture Decision Records (ADRs) documentam decisões e suas justificativas.
A IA pode gerar rascunhos excelentes quando recebe contexto suficiente. O valor
de um ADR não está apenas no registro da decisão em si, mas na documentação das
\textit{alternativas consideradas e descartadas}. Quando um novo membro da equipe
pergunta ``por que não usamos GraphQL?'', o ADR responde: ``consideramos GraphQL,
mas descartamos porque nossa equipe não tinha experiência e o prazo era de 4
semanas''. Sem o ADR, essa resposta se perde na memória institucional.

O prompt a seguir segue o formato clássico de Michael Nygard, que é o padrão
mais amplamente adotado na indústria. Cada seção do ADR tem uma função clara:
Status indica se a decisão está vigente; Contexto explica o problema; Decisão
registra a escolha; Alternativas Consideradas documenta o espaço de soluções
explorado; e Consequências tornam explícitos os impactos positivos, negativos e
os riscos residuais.

\begin{lstlisting}[language={}, caption={Prompt para geração de ADR}]
Gere um Architecture Decision Record (ADR) no formato de Michael Nygard.

Decisao: [descreva a decisao tomada]
Contexto do sistema: [descreva]
Alternativas consideradas: [liste]
Motivo da escolha: [resuma]

Formato do ADR:
# ADR-[numero]: [titulo]

## Status

\subsection{Template: Avaliação de Riscos}

Toda arquitetura tem modos de falha. A questão não é \textit{se} algo vai falhar,
mas \textit{o quê}, \textit{quando} e \textit{como} você vai saber. Este template
adota a perspectiva de um engenheiro de confiabilidade (SRE) e solicita uma
análise sistemática de modos de falha, inspirada em técnicas de FMEA (Failure
Mode and Effects Analysis) usadas na engenharia industrial.

O diferencial deste prompt é a inclusão explícita do ``mecanismo de detecção''. É
surpreendentemente comum que equipes implementem estratégias de mitigação sem
implementar mecanismos de detecção: o circuit breaker está configurado, mas
ninguém criou o alerta que avisa quando ele abre. Perguntar ``como sabemos que
falhou?'' força tanto o modelo quanto o engenheiro a pensar na observabilidade do
sistema.

\begin{lstlisting}[language={}, caption={Prompt para avaliação de riscos arquiteturais}]
Atue como engenheiro de confiabilidade (SRE) revisando a arquitetura
proposta abaixo.

[Cole ou descreva a arquitetura]

Para cada componente e integracao, identifique:
1. Modos de falha possiveis
2. Probabilidade (alta/media/baixa)
3. Impacto no usuario final (critico/degradado/imperceptivel)
4. Estrategia de mitigacao recomendada
5. Mecanismo de deteccao (como sabemos que falhou?)

Organize em uma tabela com colunas:
Componente | Modo de Falha | Probabilidade | Impacto | Mitigacao | Deteccao

Ao final, identifique os 3 maiores riscos e proponha um plano de acao.
\end{lstlisting}

\subsection{Exemplo Completo: Arquitetura de Microsserviços}

Para ilustrar como todos esses elementos se combinam, vejamos um
prompt completo para uma decisão arquitetural real. Este exemplo foi construído
com cuidado para demonstrar vários princípios simultaneamente. Primeiro, observe
como o contexto é rico e específico: não é ``uma startup'', é ``uma startup
brasileira de logística com 2 anos de mercado''. Não é ``um monolito'', é ``um
monolito em Node.js/Express com PostgreSQL que atende 500 requisições/min''. Não
é ``uma equipe'', é ``8 desenvolvedores: 2 seniores, 4 plenos, 2 juniores''.
Cada detalhe elimina ambiguidade e direciona a resposta.

Segundo, observe como os problemas reais são descritos de forma concreta: ``deploys
levam 40 minutos'', ``um bug no módulo de rastreamento derrubou o checkout'',
``dois times frequentemente geram conflitos de merge no mesmo arquivo''. Esses
são sintomas específicos que ajudam o modelo a
diagnosticar se microsserviços são realmente a solução ou se um monolito modular
seria suficiente.

Terceiro, observe a instrução final: ``Não quero uma resposta genérica sobre
microsserviços. Quero uma recomendação específica para ESTE cenário.'' Essa
meta-instrução é um mecanismo de ancoragem que combate a tendência do modelo de
recorrer a respostas enciclopédicas. Quando o modelo ``sabe'' que você não
aceitará generalidades, ele se esforça mais para contextualizar.

\begin{lstlisting}[language={}, caption={Prompt completo para decisão de arquitetura}]
Contexto: Somos uma startup brasileira de logistica com 2 anos de mercado.
Temos um monolito em Node.js/Express com PostgreSQL que atende 500
requisicoes/min. A equipe tem 8 desenvolvedores (2 seniors, 4 plenos,
2 juniors). Estamos crescendo 30% ao mes em volume de pedidos.

O monolito esta ficando dificil de manter: deploys levam 40 minutos,
um bug no modulo de rastreamento derrubou o checkout na semana passada,
e dois times diferentes frequentemente geram conflitos de merge no
mesmo arquivo.

Pergunta: devemos migrar para microsservicos agora?

Analise considerando:
1. O momento da empresa (ainda validando produto vs. escala confirmada)
2. A maturidade da equipe para operar microsservicos
3. Alternativas intermediarias (modular monolith, SOA leve)
4. Custo de migracao vs. custo de manter o monolito
5. Riscos de cada abordagem

Nao quero uma resposta generica sobre microsservicos.
Quero uma recomendacao especifica para ESTE cenario.
\end{lstlisting}

Este prompt funciona porque fornece contexto rico (tamanho da equipe, volume,
problemas reais), pede análise em eixos específicos, e explicitamente rejeita
respostas genéricas. A IA, diante dessas restrições, é forçada a produzir uma
análise contextualizada. O resultado típico inclui uma recomendação de monolito
modular como passo intermediário. Uma análise genérica sobre
microsserviços jamais produziria essa resposta, mas ela é frequentemente a mais adequada para
equipes nesse estágio de maturidade.

\section{Prompts para Geração de Código}
% ============================================================

A geração de código é o uso mais comum de LLMs entre desenvolvedores, e
também o mais propenso a resultados medíocres quando o prompt é fraco. A razão
é simples: código é preciso por natureza. Uma palavra errada em um parágrafo de
texto pode ser compreendida pelo contexto; uma linha errada em código pode
causar um bug silencioso que só se manifesta em produção semanas depois. Essa
precisão exige que os prompts para geração de código sejam igualmente precisos.

Há uma conexão forte entre prompts para geração de código e o conceito de
Design by Contract, introduzido por Bertrand Meyer. Um contrato define
pré-condições (o que deve ser verdade antes da execução), pós-condições (o que
deve ser verdade depois) e invariantes (o que deve ser verdade sempre). Um bom
prompt para geração de código é, em resumo, um contrato: define o que a
função recebe (pré-condições), o que ela retorna (pós-condições) e as restrições
que devem ser respeitadas durante toda a execução (invariantes como ``sem efeitos
colaterais'', ``thread-safe'', ``idempotente'').

\subsection{Especificando Interfaces Antes da Implementação}

O erro mais comum é pedir a implementação diretamente. O padrão correto é:
primeiro definir interfaces e contratos, depois solicitar a implementação. Esse
padrão replica a prática de engenharia de software conhecida como ``programming
to interfaces'', e por uma boa razão.

Quando você pede diretamente ``implemente um serviço de notificações'', o modelo
precisa tomar simultaneamente dezenas de decisões: quais métodos existem, quais
parâmetros cada um recebe, quais tipos são usados, como os erros são tratados,
como as dependências são injetadas. Cada decisão tomada simultaneamente pode ser
inconsistente com as demais. Quando você separa a definição de interfaces da
implementação, força o modelo a primeiro pensar na \textit{forma} do serviço,
seus contratos públicos, e só depois pensar na \textit{substância}: como
esses contratos são implementados internamente. Essa separação produz código mais
coeso, mais testável e mais consistente.

O prompt a seguir demonstra essa técnica. Observe como ele explicitamente divide
a tarefa em dois passos e solicita que o primeiro passo seja concluído ``sem
implementação''. Essa restrição é crucial: sem ela, o modelo tende a misturar
interfaces e implementação, comprometendo a clareza dos contratos.

\begin{lstlisting}[language={}, caption={Prompt para definição de interfaces antes da implementação}]
Antes de implementar, preciso definir as interfaces do modulo de
notificacoes do nosso sistema.

Requisitos:
- Suportar email, SMS e push notification
- Cada canal pode estar ativo ou inativo por usuario
- Notificacoes podem ser imediatas ou agendadas
- Precisa de retry com backoff exponencial para falhas
- Deve registrar log de todas as tentativas

Linguagem: TypeScript
Padrao: interfaces primeiro, implementacao depois

Passo 1: Defina APENAS as interfaces/tipos (sem implementacao).
Inclua:
- Interface principal do servico
- Tipos para cada canal de notificacao
- Tipo para configuracao de retry
- Tipo para o registro de log
- Enums necessarios

Passo 2: Para cada interface, adicione JSDoc explicando o contrato
(pre-condicoes, pos-condicoes, erros possiveis).
\end{lstlisting}

O que torna esse prompt poderoso é a separação explícita entre forma e substância.
As interfaces geradas podem ser revisadas, discutidas com a equipe e refinadas
antes que qualquer linha de implementação seja escrita. Se a interface está errada,
é barato mudar; se a implementação está errada porque a interface estava errada,
o custo é muito maior.

\subsection{Definindo Contratos e Tipos}

Uma vez que as interfaces estejam definidas e validadas, o próximo passo é solicitar
a implementação. Mas não qualquer implementação: uma implementação que respeite
restrições específicas de design. O prompt abaixo demonstra como fornecer essas
restrições de forma que o modelo as siga consistentemente.

A menção ao Strategy Pattern é intencional: ao nomear o padrão de design
esperado, ativamos no modelo o conhecimento sobre esse padrão e seus idiomas
característicos. Da mesma forma, a restrição de ``no máximo 15 linhas por
método'' é uma heurística que força decomposição funcional e evita os ``God
Methods'' que LLMs tendem a gerar quando não são restringidos. A exigência de
``tratamento explícito de erros (sem try/catch genérico)'' combate outra
tendência do modelo: envolver tudo em try/catch com uma mensagem genérica de
erro, que em produção torna o debugging impossível.

\begin{lstlisting}[language={}, caption={Prompt para geração de código com contratos tipados}]
Implemente o servico de NotificationService seguindo estas interfaces:

[cole as interfaces definidas no passo anterior]

Restricoes de implementacao:
- Use injecao de dependencia (construtor)
- Cada canal deve ser uma Strategy (Strategy Pattern)
- O retry deve usar backoff exponencial: 1s, 2s, 4s, 8s, max 3 tentativas
- Funcoes puras onde possivel; efeitos colaterais isolados
- Tratamento explicito de erros (sem try/catch generico)
- Cada metodo deve ter no maximo 15 linhas

Para cada metodo implementado, adicione um comentario de uma linha
explicando a decisao de design.
\end{lstlisting}

\subsection{Solicitando Código com Testes}

Código sem testes é código sem garantia. Quando solicitamos à IA que gere código
e testes simultaneamente, estamos aplicando uma forma de TDD (Test-Driven
Development) assistida: os testes definem o comportamento esperado, e o código é
gerado para satisfazê-los. Os testes servem também como
\textit{documentação executável} do prompt, pois demonstram, de forma
verificável, que o código faz o que foi pedido.

O prompt a seguir é um exemplo de especificação completa. Observe como as regras
de negócio são descritas com valores numéricos concretos, não com descrições
vagas. ``Peso até 500g: R\$ 8,90 (fixo)'' é inequívoco; ``frete proporcional ao
peso'' é ambíguo. A seção de testes é igualmente precisa, solicitando boundary
values (valores nos limites das faixas) e edge cases (valores extremos como 0g).
A instrução ``não use mocks'' e ``a função deve ser pura'' garantem que os testes
são determinísticos e rápidos, sem dependências externas.

\begin{lstlisting}[language={}, caption={Prompt para geração de código com testes}]
Implemente a funcao de calculo de frete com as seguintes regras:

Regras de negocio:
- Peso ate 500g: R$ 8,90 (fixo)
- Peso de 501g a 2kg: R$ 15,90 + R$ 2,00 por kg adicional
- Peso acima de 2kg: R$ 25,90 + R$ 3,50 por kg adicional
- Frete gratis para pedidos acima de R$ 200,00
- CEPs da regiao Sul: desconto de 10%
- CEPs invalidos devem lancar erro especifico

Linguagem: Python 3.12
Framework de teste: pytest

Entregue:
1. A funcao calcular_frete(peso_gramas, valor_pedido, cep)
2. Testes unitarios cobrindo:
   - Cada faixa de peso (boundary values)
   - Frete gratis
   - Desconto regional
   - CEP invalido
   - Valores limites (edge cases): 0g, 500g, 501g, 2000g, 2001g
3. Docstring com exemplos (doctest)

Nao use mocks. A funcao deve ser pura (sem I/O).
\end{lstlisting}

A eficácia desse prompt pode ser verificada de forma objetiva: execute os testes
gerados e veja quantos passam. Em nossa experiência, prompts com esse nível de
especificidade produzem testes que passam em 85-95\% dos casos na primeira
tentativa, contra menos de 50\% para prompts vagos como ``implemente cálculo de
frete com testes''.

\subsection{Pedindo Explicações com Código}

Código que funciona mas que ninguém entende é uma bomba-relógio. Quando solicitamos
à IA que explique suas decisões junto com o código, estamos produzindo dois
artefatos em um: a implementação e sua documentação de design. A explicação serve
tanto para validar que o modelo tomou decisões sensatas quanto para servir de
referência futura para quem for manter o código.

O prompt abaixo pede três tipos de comentário para cada bloco de código: o que
faz (descrição funcional), por que essa abordagem (justificativa de design), e
qual alternativa existiria (consciência de trade-offs). Essa tríade é inspirada
na prática de ``decisão de design'' usada em revisões de arquitetura: para cada
escolha, registrar a escolha, a justificativa e as alternativas descartadas.
A seção final ``Como Usar'' é um padrão de documentação pragmática que
transforma o código de um artefato isolado em um componente integrável.

\begin{lstlisting}[language={}, caption={Prompt para código explicado}]
Implemente um rate limiter usando o algoritmo Token Bucket em Go.

Requisitos:
- Thread-safe (sera usado em servidor HTTP concorrente)
- Configuravel: tokens por segundo e burst maximo
- Interface simples: Allow() bool

Para cada bloco de codigo (5-10 linhas), adicione um comentario
explicando:
1. O QUE o bloco faz
2. POR QUE esta abordagem foi escolhida
3. Qual ALTERNATIVA existiria e por que nao foi usada

Ao final, adicione uma secao "## Como Usar" com exemplo de
integracao em um middleware HTTP.
\end{lstlisting}

\section{Prompts para Revisão de Código}
% ============================================================

A IA como revisora de código funciona melhor quando recebe critérios explícitos.
Sem critérios, ela tende a fazer comentários superficiais (``considere adicionar
tratamento de erro'') em vez de análises profundas. Esse comportamento não é
um bug do modelo; é uma consequência do treinamento. O modelo foi treinado
em milhões de code reviews, e a maioria dos comentários de code review que
existem na internet são, infelizmente, superficiais. Para obter revisões
profundas, é preciso instruir explicitamente o modelo a ir além do óbvio.

Há um paralelo instrutivo com a prática de code review entre humanos. Um reviewer
sem checklist tende a focar no que salta aos olhos: nomes de variáveis,
formatação, o bug mais óbvio. Um reviewer com checklist percorre sistematicamente
cada aspecto: segurança, performance, tratamento de erros, aderência a padrões,
testabilidade. Prompts para revisão de código funcionam da mesma forma: o checklist
explícito no prompt substitui o checklist mental do reviewer experiente.

A estratégia mais eficaz é usar revisões \textbf{focadas}: em vez de pedir
``revise este código'', peça uma revisão focada em um aspecto específico. Uma
revisão focada em bugs produz resultados diferentes de uma revisão focada em
segurança, que produz resultados diferentes de uma revisão focada em design.
Revisões genéricas tendem a ser rasas em todos os aspectos; revisões focadas
tendem a ser profundas no aspecto escolhido.

\subsection{Prompt para Encontrar Bugs}

O prompt a seguir é um exemplo de revisão focada. A instrução ``EXCLUSIVAMENTE por
bugs'' é uma restrição poderosa: ela impede que o modelo desvie para sugestões de
estilo ou refatoração, que são válidas mas fora do escopo da tarefa. A exigência
de ``cenário que dispara o bug (input específico)'' é inspirada na prática de
escrever testes de regressão: um bug que não pode ser reproduzido por um input
específico provavelmente não é um bug real. A instrução final, ``se não
encontrar bugs, diga explicitamente'', combate a tendência do modelo de
inventar problemas para parecer útil.

\begin{lstlisting}[language={}, caption={Prompt para análise de bugs}]
Revise o codigo abaixo procurando EXCLUSIVAMENTE por bugs.
Nao sugira melhorias de estilo ou refatoracoes --- apenas bugs reais
que causariam comportamento incorreto em producao.

[cole o codigo]

Para cada bug encontrado:
1. Linha(s) afetada(s)
2. Descricao do bug em uma frase
3. Cenario que dispara o bug (input especifico)
4. Correcao sugerida (diff minimo)
5. Severidade: critica / alta / media / baixa

Se nao encontrar bugs, diga explicitamente "Nenhum bug identificado"
em vez de inventar problemas inexistentes.
\end{lstlisting}

\subsection{Prompt para Análise de Segurança}

Segurança é um domínio onde revisão automatizada pode ser extraordinariamente
valiosa, porque vulnerabilidades de segurança seguem padrões bem catalogados
(OWASP Top 10, CWE) que LLMs conhecem em profundidade. O prompt abaixo lista
explicitamente as categorias de vulnerabilidade a verificar, evitando que o
modelo foque apenas nas mais óbvias (SQL injection) e ignore as mais sutis
(race conditions que afetam segurança, deserialização insegura).

A classificação usando OWASP Top 10 e CVSS simplificado transforma a saída de
uma lista de ``problemas'' em uma avaliação de risco priorizada, permitindo que a
equipe decida quais vulnerabilidades corrigir primeiro. O formato de tabela
facilita a integração com ferramentas de tracking e a criação de tickets.

\begin{lstlisting}[language={}, caption={Prompt para análise de segurança}]
Atue como engenheiro de seguranca realizando uma revisao de codigo
focada em vulnerabilidades.

[cole o codigo]

Verifique especificamente:
1. Injecao (SQL, NoSQL, OS command, LDAP)
2. Cross-Site Scripting (XSS) --- stored e reflected
3. Autenticacao e autorizacao quebradas
4. Exposicao de dados sensiveis (logs, respostas de API)
5. Deserializacao insegura
6. Dependencias com vulnerabilidades conhecidas
7. Criptografia fraca ou mal implementada
8. Race conditions que afetem seguranca

Para cada vulnerabilidade:
- Classifique usando OWASP Top 10 (categoria)
- Estime CVSS simplificado (1-10)
- Mostre o codigo vulneravel e o codigo corrigido
- Explique o vetor de ataque em linguagem simples

Formato: tabela Markdown com colunas:
Vulnerabilidade | OWASP | CVSS | Linha | Correcao
\end{lstlisting}

\subsection{Prompt para Melhoria de Design}

A terceira dimensão de revisão é o design. Enquanto revisão de bugs pergunta ``isso
funciona corretamente?'' e revisão de segurança pergunta ``isso é seguro?'',
revisão de design pergunta ``isso é bem projetado?''. A premissa do prompt abaixo
é explícita: ``assuma que o código funciona corretamente''. Essa premissa libera o
modelo para focar inteiramente em questões estruturais: coesão, acoplamento,
testabilidade, extensibilidade e legibilidade.

A referência a princípios nomeados (SOLID, KISS, DRY) é intencional: ao pedir
que o modelo cite o princípio violado, forçamos uma justificativa fundamentada
em vez de opiniões subjetivas. ``Este método viola o Single Responsibility
Principle porque faz validação e persistência'' é uma observação acionável;
``este método é meio grande'' é uma observação vaga. A priorização por impacto
na manutenibilidade é o mecanismo que transforma uma lista de ``seria legal
mudar'' em uma lista de ``precisamos mudar, na ordem certa''.

\begin{lstlisting}[language={}, caption={Prompt para sugestões de melhoria de design}]
Revise o codigo abaixo do ponto de vista de DESIGN, nao de bugs.
Assuma que o codigo funciona corretamente.

[cole o codigo]

Avalie nos seguintes eixos:
1. Coesao: cada classe/funcao tem uma unica responsabilidade?
2. Acoplamento: ha dependencias desnecessarias entre modulos?
3. Testabilidade: e facil escrever testes unitarios?
4. Extensibilidade: adicionar um novo caso requer mudar codigo existente?
5. Legibilidade: um dev pleno entenderia em 5 minutos?

Para cada problema encontrado:
- Cite o principio violado (SOLID, KISS, DRY, etc.)
- Mostre o antes e o depois (refatoracao minima)
- Explique o beneficio da mudanca

Priorize: liste primeiro os problemas com maior impacto na
manutenibilidade.
\end{lstlisting}

\section{Prompts para Refatoração}
% ============================================================

Refatorar com IA exige cuidado especial: o comportamento deve ser preservado.
Prompts vagos (``melhore este código'') geram mudanças que parecem melhores
mas podem quebrar contratos implícitos. A definição clássica de refatoração,
cunhada por Martin Fowler, é ``uma mudança na estrutura interna do software
que melhora sua compreensibilidade e reduz seu custo de modificação, sem alterar
seu comportamento observável''. A palavra-chave é \textit{sem alterar seu
comportamento observável}. Quando pedimos à IA que ``melhore'' código sem essa
restrição explícita, frequentemente obtemos código que parece melhor mas se
comporta diferente em casos de borda. Isso gera uma regressão silenciosa que pode levar
semanas para ser detectada.

A estratégia fundamental para refatoração assistida por IA é a mesma da
refatoração manual: passos pequenos e verificáveis. Cada passo deve ser uma
transformação atômica com comportamento preservado. O problema de pedir
``refatore tudo de uma vez'' é que o modelo precisa fazer múltiplas
transformações simultaneamente, aumentando exponencialmente a chance de
introduzir regressões. Pedir uma transformação por vez (``extraia o método
X'', depois ``mova a classe Y'', depois ``elimine o parâmetro Z'') mantém
cada passo verificável e reversível.

\subsection{Identificando Code Smells}

Antes de refatorar, é preciso identificar \textit{o que} refatorar. O catálogo
de code smells de Martin Fowler fornece um vocabulário compartilhado para nomear
problemas de design. Usar esse vocabulário no prompt é importante por duas razões.
Primeiro, ativa no modelo o conhecimento específico sobre cada smell e suas
técnicas de refatoração recomendadas. Segundo, produz uma saída que usa a mesma
terminologia que sua equipe, facilitando a comunicação e a priorização.

O prompt abaixo limita a saída a no máximo sete smells, focando nos mais
impactantes. Sem essa limitação, o modelo tende a produzir listas exaustivas
onde problemas triviais se misturam com problemas críticos, dificultando a
priorização. A classificação de prioridade em três níveis (afeta bugs, afeta
manutenção, afeta legibilidade) permite que a equipe decida onde investir
esforço de refatoração.

\begin{lstlisting}[language={}, caption={Prompt para identificação de code smells}]
Analise o codigo abaixo e identifique code smells conforme o catalogo
de Martin Fowler (Refactoring, 2nd Edition).

[cole o codigo]

Para cada smell encontrado:
1. Nome do smell (ex: Long Method, Feature Envy, Data Clump)
2. Localizacao no codigo (linhas)
3. Por que e um problema neste contexto
4. Refatoracao recomendada (nome da tecnica de Fowler)
5. Prioridade: alta (afeta bugs), media (afeta manutencao),
   baixa (afeta legibilidade)

Nao liste mais de 7 smells. Foque nos mais impactantes.
\end{lstlisting}

\subsection{Refatorações Específicas}

Quando o code smell foi identificado e a técnica de refatoração escolhida, o
próximo passo é executar a refatoração. O prompt abaixo demonstra como solicitar
uma refatoração específica de forma segura. A técnica ``Replace Conditional with
Polymorphism'' é uma das mais poderosas do catálogo de Fowler: transforma cascatas
de if/else ou switch/case em hierarquias de classes com comportamento polimórfico,
tornando o código mais extensível (adicionar um novo caso é adicionar uma nova
classe, não editar um switch existente) e mais testável (cada classe pode ser
testada isoladamente).

A restrição ``o comportamento externo deve ser IDÊNTICO'' é a âncora de
segurança do prompt. A exigência de ``teste que prova equivalência de
comportamento'' é a verificação objetiva dessa âncora: se os mesmos inputs
produzem os mesmos outputs antes e depois da refatoração, o comportamento foi
preservado.

\begin{lstlisting}[language={}, caption={Prompt para refatoração dirigida}]
Refatore o codigo abaixo aplicando EXCLUSIVAMENTE a tecnica
"Replace Conditional with Polymorphism".

[cole o codigo com switch/case ou if/else longo]

Restricoes:
- O comportamento externo deve ser IDENTICO (mesmos inputs = mesmos outputs)
- Mantenha os mesmos nomes de metodos publicos
- Adicione interfaces/tipos para o polimorfismo
- Cada classe resultante deve ter no maximo 30 linhas

Entregue:
1. Codigo refatorado completo
2. Teste que prova equivalencia de comportamento (antes e depois
   produzem os mesmos resultados para os mesmos inputs)
3. Explicacao de 3 frases sobre por que esta refatoracao melhora
   o design
\end{lstlisting}

\subsection{Mantendo Comportamento Durante Refatoração}

A regra de ouro da refatoração é: mudanças de estrutura não devem mudar
comportamento. Ao usar IA para refatorar, sempre exija evidência de que o
comportamento foi preservado. Essa exigência não é paranoia. É engenharia
disciplinada.

Testes de equivalência são a forma mais robusta de garantir preservação de
comportamento. Solicite que a IA gere testes que rodem contra a versão original
e a versão refatorada, verificando que ambas produzem os mesmos resultados para os
mesmos inputs. Esses testes funcionam como uma rede de segurança: se algum teste
falhar após a refatoração, você sabe imediatamente que algo quebrou, e pode
reverter a mudança antes que ela se propague.

Refatorar em passos pequenos é igualmente importante. Em vez de solicitar
``refatore tudo'', peça uma refatoração por vez: ``extraia o método
\texttt{calcularDesconto}'', depois ``mova a classe \texttt{ValidadorCEP} para
seu próprio arquivo'', depois ``elimine o parâmetro \texttt{debug} que não é
mais usado''. Cada passo pode ser verificado isoladamente, e se algo der errado,
é fácil identificar qual passo introduziu o problema. Essa abordagem incremental
é mais lenta mas exponencialmente mais segura.

Usar o compilador como aliado é outra estratégia valiosa. Peça código que compile
sem warnings. Se compila e os testes passam, a chance de regressão comportamental
é significativamente menor. Note que compilação sem warnings não é garantia de
comportamento preservado; é uma condição necessária, mas não suficiente. Os
testes de equivalência complementam o que o compilador não pode verificar.

Desconfie de refatorações que mudam assinaturas públicas. Se a API
pública mudou (novos parâmetros, tipos de retorno diferentes, métodos
renomeados), não é refatoração, é redesign. Redesign é legítimo e muitas vezes
necessário, mas deve ser tratado de forma diferente: com migração planejada,
versionamento de API e comunicação com os consumidores. Quando o modelo muda
uma assinatura pública durante uma ``refatoração'', isso é um sinal de alerta
que merece investigação.

% ============================================================
\section{Padrões Avançados de Prompting}
% ============================================================

Para além dos prompts diretos, existem técnicas avançadas que melhoram
significativamente a qualidade das respostas para problemas complexos. Essas
técnicas exploram propriedades fundamentais de como os LLMs processam texto,
e entendê-las permite aplicá-las de forma mais eficaz.

\subsection{Chain of Thought (CoT) para Arquitetura}

Chain of Thought força a IA a ``mostrar o trabalho'', raciocinar passo a
passo antes de apresentar a conclusão. Para decisões arquiteturais, isso é
especialmente valioso porque expõe premissas que podem estar erradas.

A ciência cognitiva por trás do Chain of Thought é fascinante e instrutiva.
Quando humanos enfrentam problemas complexos, a resolução passo a passo é
consistentemente superior à intuição imediata. O psicólogo Daniel Kahneman
distingue dois modos de pensamento: o Sistema 1 (rápido, intuitivo, propenso a
vieses) e o Sistema 2 (lento, deliberado, analítico). Chain of Thought é,
por definição, uma técnica para forçar o Sistema 2 --- tanto em humanos quanto
em LLMs.

Para LLMs, a mecânica é concreta. Cada token gerado pelo modelo é condicionado
por todos os tokens anteriores. Quando o modelo escreve um passo intermediário de
raciocínio (``Primeiro, os requisitos indicam que precisamos de baixa
latência...''), esse texto passa a fazer parte do contexto para os tokens
seguintes. O modelo literalmente ``lê'' seu próprio raciocínio e o usa para
informar o próximo passo. Sem Chain of Thought, o modelo tenta ``pular''
diretamente para a conclusão, perdendo os passos intermediários que fundamentam
essa conclusão. É como pedir a um estudante que resolva uma integral complexa
mentalmente: mesmo que chegue à resposta certa, a chance de erro é muito maior do
que se escrevesse cada passo no papel.

Em decisões arquiteturais, o Chain of Thought é particularmente valioso porque
expõe o raciocínio do modelo de forma auditável. Se o modelo conclui que
``Elasticsearch é a melhor opção'', mas no passo 2 assume que ``a equipe tem
experiência com a Elastic Stack'' (quando na verdade não tem), você pode
identificar e corrigir essa premissa falsa antes que ela contamine a conclusão.
Sem Chain of Thought, você receberia apenas a conclusão e teria que adivinhar
por que o modelo a escolheu.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.85\textwidth]{imagens/05-prompts-arquitetura-codigo-01-chain-of-thought-para-decisao-arquitetural.png}
    \caption{Chain of Thought para decisão arquitetural}
\end{figure}

O prompt a seguir demonstra Chain of Thought aplicado a uma decisão concreta.
Observe como cada passo é numerado e tem um escopo claro. O passo 1 lista
requisitos (explícitos e implícitos), o passo 2 identifica restrições, o passo 3
enumera alternativas, o passo 4 analisa cada alternativa nos mesmos critérios, e
o passo 5 sintetiza uma recomendação. A instrução final (``mostre seu
raciocínio completo em cada passo, não pule para a conclusão'') é a ancoragem
que garante que o modelo não tentará atalhar o processo.

\begin{lstlisting}[language={}, caption={Prompt usando Chain of Thought}]
Preciso decidir como implementar busca textual no nosso sistema de
e-commerce (catalogo de 500k produtos, em portugues).

Pense passo a passo:

1. Primeiro, liste os requisitos implicitos e explicitos de uma
   busca textual em e-commerce (relevancia, velocidade, typos,
   sinonimos, filtros)
2. Depois, identifique as restricoes do nosso cenario
   (equipe de 3 devs, stack Python/Django, budget limitado)
3. Entao, enumere pelo menos 4 alternativas tecnicas
   (Elasticsearch, PostgreSQL full-text, Typesense, Algolia, etc.)
4. Para cada alternativa, analise: custo, complexidade, qualidade
   de busca em portugues, facilidade de operacao
5. Finalmente, recomende UMA opcao com justificativa clara

Mostre seu raciocinio completo em cada passo. Nao pule para a
conclusao.
\end{lstlisting}

A análise do que torna esse prompt eficaz revela vários mecanismos operando
simultaneamente. A estrutura numerada impõe uma sequência lógica. A instrução
``primeiro liste requisitos'' antes de ``enumere alternativas'' garante que as
alternativas serão avaliadas contra critérios previamente definidos, e não
escolhidas arbitrariamente. A restrição ``pelo menos 4 alternativas'' combate o
viés de ancoragem. E a instrução final transforma o Chain of Thought de
sugestão em exigência.

\subsection{Tree of Thoughts para Design Complexo}

Enquanto Chain of Thought é linear, Tree of Thoughts explora múltiplos caminhos
simultaneamente e seleciona o melhor. É útil para problemas onde a primeira
abordagem que vem à mente raramente é a melhor. Se Chain of Thought é caminhar
por uma trilha passo a passo, Tree of Thoughts é explorar três trilhas diferentes
e, ao final, comparar os destinos e escolher o melhor percurso.

A técnica é inspirada em como especialistas humanos abordam problemas de design
difíceis. Um arquiteto experiente não se compromete com a
primeira solução que imagina; ele deliberadamente explora alternativas,
desenvolve cada uma até o ponto em que seus problemas ficam evidentes, e só então
faz uma escolha informada. Tree of Thoughts replica esse processo no LLM,
instruindo o modelo a desenvolver cada caminho ``completamente'' antes de compará-
los.

O prompt a seguir aplica Tree of Thoughts ao design de um sistema de permissões,
um problema notoriamente complexo onde a escolha errada gera dívida técnica por
anos. Cada ``caminho'' é desenvolvido independentemente, com seus pontos fortes
e fracos. A avaliação final, que compara os caminhos nos mesmos eixos, permite
uma decisão fundamentada.

\begin{lstlisting}[language={}, caption={Prompt usando Tree of Thoughts}]
Preciso projetar o sistema de permissoes do nosso SaaS multi-tenant.

Explore 3 caminhos de design diferentes:

Caminho 1: RBAC (Role-Based Access Control)
- Desenvolva este caminho completamente
- Identifique onde ele falha para nosso caso

Caminho 2: ABAC (Attribute-Based Access Control)
- Desenvolva este caminho completamente
- Identifique onde ele falha para nosso caso

Caminho 3: Uma abordagem hibrida
- Desenvolva este caminho completamente
- Identifique onde ele falha para nosso caso

Para cada caminho, avalie:
- Simplicidade de implementacao
- Flexibilidade para requisitos futuros
- Performance com 10k usuarios e 500 permissoes
- Facilidade de auditoria

Depois de explorar os 3 caminhos, selecione o melhor e justifique
a escolha com base na avaliacao.
\end{lstlisting}

\subsection{Refinamento Iterativo}

Raramente o primeiro prompt produz o resultado perfeito. O refinamento iterativo
é uma técnica onde cada prompt subsequente corrige e aprofunda o anterior. É o
equivalente em prompting ao desenvolvimento iterativo em software: em vez de
tentar acertar tudo na primeira tentativa, você entrega incrementalmente,
validando a cada iteração.

A mecânica do refinamento iterativo segue uma progressão natural. O primeiro
prompt solicita a visão geral em alto nível: a arquitetura conceitual, os
componentes principais, os fluxos de dados. Esse resultado é revisado pelo
engenheiro, que identifica quais áreas precisam de aprofundamento. O segundo
prompt solicita detalhamento de um componente específico, referenciando
explicitamente a visão geral do prompt anterior. O terceiro prompt solicita
análise de riscos do componente detalhado, novamente referenciando os resultados
anteriores. O quarto prompt solicita a implementação, com base em toda a
fundamentação construída nos prompts anteriores.

O princípio fundamental é que cada prompt deve referenciar e construir sobre a
resposta anterior, nunca recomeçar do zero. Recomeçar do zero desperdiça o
contexto acumulado e pode produzir respostas inconsistentes entre si. Referenciar
explicitamente (``com base na arquitetura que definimos, onde o serviço de
pagamentos se comunica com o serviço de notificações via fila SQS'') ancora o
modelo no contexto correto e garante continuidade.

O refinamento iterativo é também uma ferramenta de \textbf{aprendizado}. A cada
iteração, o engenheiro refina não apenas o resultado, mas sua própria compreensão
do problema. Perguntas que surgem durante a revisão do primeiro resultado
(``e se o serviço de notificações ficar indisponível?'') revelam requisitos que
não estavam explícitos no prompt original. O refinamento iterativo é, portanto,
um processo de co-descoberta entre o engenheiro e a IA.

\subsection{Prompt Chaining para Tarefas Complexas}

Prompt chaining decompõe uma tarefa complexa em subtarefas sequenciais, onde a
saída de um prompt alimenta o próximo. Enquanto refinamento iterativo é sobre
aprofundar progressivamente, prompt chaining é sobre decompor horizontalmente:
dividir uma tarefa grande em etapas menores e executá-las em sequência.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.85\textwidth]{imagens/05-prompts-arquitetura-codigo-02-workflow-de-prompt-engineering.png}
    \caption{Workflow de Prompt Engineering}
\end{figure}

A analogia com pipelines de software é direta. Assim como um pipeline de CI/CD
decompõe o processo de entrega em etapas (build, test, lint, deploy), prompt
chaining decompõe uma tarefa de desenvolvimento em etapas (especificar, implementar,
testar, revisar, empacotar). Cada etapa produz um artefato que é input da
próxima. E, assim como em um pipeline, se uma etapa falha, o feedback é imediato
e localizado: sabemos exatamente qual passo precisa ser refeito.

Considere o exemplo prático de criar um microsserviço completo via prompt
chaining. O primeiro prompt define interfaces e contratos do serviço de
pagamentos, produzindo como saída interfaces TypeScript validáveis. O segundo
prompt recebe essas interfaces e solicita a implementação usando Clean
Architecture, produzindo o código do serviço. O terceiro prompt recebe o código
e solicita testes unitários abrangentes. O quarto prompt recebe tanto o código
quanto os testes e solicita uma revisão buscando bugs e gaps de cobertura. O
quinto prompt solicita Dockerfile e docker-compose para o serviço.

Cada passo produz um artefato validável. Se o terceiro prompt (testes) revela que
o código do segundo prompt (implementação) é difícil de testar, isso é feedback
direto e acionável para refatorar antes de avançar. Essa dinâmica de feedback é o
que torna prompt chaining superior a um único prompt monolítico que tenta produzir
tudo de uma vez: o prompt monolítico não tem mecanismos de verificação
intermediária, e quando o resultado final tem problemas, é difícil identificar em
qual ``etapa mental'' o modelo errou.

\subsection{System Prompts e Personalização}

Muitas ferramentas de IA permitem configurar um \textit{system prompt}, uma
instrução persistente que molda todas as interações. Para engenheiros de
software, um bom system prompt padroniza a qualidade das respostas e elimina a
necessidade de repetir as mesmas instruções em cada prompt individual.

O system prompt funciona como uma ``configuração de ambiente'' para o modelo. Da
mesma forma que um desenvolvedor configura seu editor, seu linter e seu
formatter para impor um padrão de qualidade em todo o código que escreve, o system
prompt configura o modelo para impor um padrão de qualidade em todas as respostas
que gera. Um system prompt bem elaborado transforma um modelo genérico em um
assistente especializado que já ``sabe'' suas preferências, sua stack, seus
princípios de design e seus padrões de resposta.

O exemplo a seguir é um system prompt projetado para engenharia de software. Cada
bloco foi cuidadosamente construído. O bloco de ``princípios'' define os valores
que devem guiar todas as respostas. São como os princípios de design de um
sistema. O bloco ``quando eu pedir código'' define as convenções técnicas. O bloco
``quando eu pedir arquitetura'' define as premissas de contexto que o modelo deve
assumir quando não são explicitadas. O bloco ``quando eu pedir revisão'' define o
estilo de feedback esperado.

\begin{lstlisting}[language={}, caption={Exemplo de system prompt para engenharia de software}]
Voce e um engenheiro de software senior com 15 anos de experiencia.

Principios que voce segue:
- Simplicidade sobre complexidade (KISS)
- Codigo testavel por padrao
- Erros explicitos, nunca silenciosos
- Imutabilidade quando possivel
- Nomes descritivos, nunca abreviacoes

Quando eu pedir codigo:
- Use TypeScript com strict mode
- Inclua tipos explicitos (sem 'any')
- Adicione tratamento de erro
- Sugira testes relevantes
- Comente decisoes nao obvias, nao o obvio

Quando eu pedir arquitetura:
- Considere equipe de 5-8 devs
- Assuma cloud AWS
- Priorize operabilidade sobre elegancia
- Sempre mencione trade-offs

Quando eu pedir revisao:
- Seja direto e especifico
- Cite principios quando relevante
- Sugira correcao, nao apenas aponte problema
- Priorize por impacto
\end{lstlisting}

\subsection{Estratégias de Conversa Multi-turno}

Em conversas longas com LLMs, o contexto se degrada. Essa degradação é uma
consequência da janela de contexto finita e do mecanismo de atenção: à medida
que a conversa cresce, tokens mais antigos recebem menos ``atenção'' e podem ser
efetivamente ``esquecidos'' pelo modelo. Para conversas técnicas complexas que se
estendem por muitos turnos, existem técnicas específicas para mitigar essa
degradação.

Resumos periódicos são a técnica mais fundamental. A cada cinco ou dez turnos,
solicite que a IA resuma o que foi decidido até o momento. Esse resumo serve como
um ``checkpoint'' da conversa: ele consolida as decisões, premissas e artefatos
produzidos em um bloco compacto que pode ser usado como contexto para os próximos
prompts. É como fazer commits frequentes em vez de acumular mudanças sem salvar.
Se o contexto se degradar, você tem um ponto de recuperação recente.

Referência explícita é a técnica complementar. Em vez de depender da memória
implícita do modelo sobre o que foi discutido três turnos atrás, referencie
explicitamente: ``com base na arquitetura que definimos no turno 3, onde o serviço
de pagamentos se comunica com o serviço de notificações via fila SQS, agora
implemente o módulo X''. Essa referência explícita reativa o contexto relevante
e impede que o modelo ``esqueça'' decisões anteriores.

Contexto incremental é a abordagem que equilibra eficiência e completude. Em cada
turno, adicione informação nova sem repetir tudo o que já foi dito. Porém,
reafirme restrições críticas que não podem ser esquecidas: se a latência P99 deve
ser menor que 100ms, repita isso em cada prompt onde a performance é relevante.
O custo em tokens de repetir uma restrição crítica é infinitamente menor que o
custo de refazer um artefato porque o modelo ``esqueceu'' a restrição.

Checkpoints de validação são a técnica mais subestimada. Periodicamente, peça à
IA para listar todas as premissas que está assumindo. A resposta frequentemente
revela premissas que o modelo ``inferiu'' e que estão incorretas. Por exemplo,
ele pode assumir que o sistema usa autenticação baseada em sessão quando na verdade usa JWT,
ou assumir que o banco de dados é relacional quando na verdade é um
documento store. Corrigir essas premissas antes de prosseguir evita retrabalho
significativo nos turnos subsequentes.

\section{Anti-padrões de Prompting}
% ============================================================

Conhecer o que \textbf{não} fazer é tão importante quanto saber o que fazer.
Anti-padrões de prompting desperdiçam tempo e geram falsa confiança. A falsa
confiança é especialmente perigosa: quando o prompt é ruim e a resposta parece
boa, o engenheiro pode aceitar código ou recomendações fundamentalmente
inadequadas, confiando na fluência superficial do modelo.

\subsection{Catálogo de Anti-padrões}

\subsection{Anti-padrão 1: Prompts Vagos}

O anti-padrão mais comum e mais destrutivo. ``Faça X'' sem especificar
\textbf{como}, \textbf{por quê}, \textbf{para quem}, e \textbf{com quais
restrições}.

Imagine a história de João, um desenvolvedor pleno que acaba de receber a tarefa
de criar uma API de usuários para o novo sistema da empresa. João abre o ChatGPT
e digita: ``Crie uma API REST de usuários''. A resposta vem em segundos: um CRUD
básico em Express.js com MongoDB, sem validação, sem autenticação, sem tratamento
de erro, sem testes. João olha o código, pensa ``parece bom'', copia para o
projeto, adapta algumas coisas, e faz o commit. Duas semanas depois, o time de
segurança encontra que a API aceita emails duplicados, não valida o formato do
CEP, armazena senhas em texto puro e retorna stack traces completas em erros 500.
O tempo gasto corrigindo esses problemas foi cinco vezes maior do que o tempo que
João ``economizou'' usando a IA.

Agora imagine a mesma tarefa com um prompt específico. Maria, outra desenvolvedora
da mesma equipe, recebe a mesma tarefa. Ela gasta dez minutos elaborando um
prompt que especifica cada endpoint, cada campo com suas validações, o padrão de
soft delete, a stack exata do projeto (Python 3.12, FastAPI, SQLAlchemy 2.0,
PostgreSQL), o padrão arquitetural (Repository Pattern + Service Layer), o
framework de validação (Pydantic v2) e a exigência de testes com pytest. O
código gerado já inclui validações, tratamento de erro, soft delete e testes.
Maria revisa, faz dois ajustes menores, e o código está pronto para code review.
O tempo total, incluindo a elaboração do prompt, foi menor do que o tempo de
João, e o resultado foi incomparavelmente superior.

\begin{lstlisting}[language={}, caption={Anti-padrão: prompt vago vs. prompt específico}]
--- RUIM ---
Crie uma API REST de usuarios.

--- BOM ---
Crie uma API REST de usuarios com os seguintes endpoints:

POST /users - Criar usuario
  - Campos: nome (string, 2-100 chars), email (unico, validado),
    senha (min 8 chars, 1 maiuscula, 1 numero)
  - Retorno: 201 com usuario criado (sem senha) ou 422 com erros

GET /users/:id - Buscar usuario
  - Retorno: 200 com usuario ou 404

PUT /users/:id - Atualizar usuario
  - Apenas campos enviados sao atualizados (patch semantics)
  - Retorno: 200 com usuario atualizado

DELETE /users/:id - Soft delete
  - Marca deleted_at, nao remove do banco
  - Retorno: 204

Stack: Python 3.12, FastAPI, SQLAlchemy 2.0, PostgreSQL
Padrao: Repository Pattern + Service Layer
Validacao: Pydantic v2
Testes: pytest com fixtures para cada endpoint
\end{lstlisting}

A diferença entre os dois prompts é a diferença entre uma conversa de corredor e
uma especificação técnica. A primeira produz ``algo''; a segunda produz o que
você precisa.

\subsection{Anti-padrão 2: Contexto Implícito}

Assumir que a IA ``sabe'' sobre seu projeto, sua equipe, suas convenções é um
dos erros mais comuns, e mais compreensíveis. Quando trabalhamos com colegas
humanos que compartilham nosso contexto há meses ou anos, podemos dizer ``adicione
autenticação no nosso backend'' e eles sabem exatamente o que isso significa:
qual framework, qual banco, qual padrão de middleware, qual convenção de nomes. A
IA não tem esse contexto compartilhado.

Considere a história de Carlos, um tech lead que vem usando o copiloto de IA há
meses e desenvolveu o hábito de conversar com ele como conversa com sua equipe.
Carlos digita: ``Adicione autenticação no nosso backend.'' O modelo gera
autenticação baseada em sessão com Express.js e MongoDB. É uma
resposta genérica razoável, mas completamente inadequada para o backend de Carlos, que usa
FastAPI com PostgreSQL e já tem um padrão de middleware definido. Carlos gasta
trinta minutos adaptando o código gerado para sua stack, lutando contra decisões
de design que não se encaixam. Se tivesse dedicado dois minutos a fornecer
contexto (``adicione autenticação JWT ao backend FastAPI existente, usando
PostgreSQL, SQLAlchemy ORM, e seguindo o padrão de middleware que temos: [exemplo
do middleware existente]''), o resultado teria sido utilizável desde o início.

O antídoto para o contexto implícito é a disciplina de explicitação. Antes de
cada prompt, pergunte-se: ``se eu entregasse este prompt a um desenvolvedor que
nunca viu meu projeto, ele teria informação suficiente para fazer o que eu
preciso?'' Se a resposta for não, adicione o contexto que falta. Isso inclui:
a stack tecnológica, os padrões arquiteturais adotados, as convenções de
nomenclatura, exemplos de código existente que demonstrem o estilo do projeto, e
qualquer restrição de negócio ou regulatória relevante.

\subsection{Anti-padrão 3: Aceitar a Primeira Resposta}

A primeira resposta da IA é um rascunho, não um produto final. Esse é talvez o
anti-padrão mais perigoso, porque explora uma vulnerabilidade cognitiva humana: a
\textit{automation bias}, a tendência de confiar excessivamente em resultados
gerados por sistemas automatizados.

A história de Ana ilustra o perigo. Ana é uma desenvolvedora sênior trabalhando
em um sistema financeiro. Ela pede ao modelo que implemente a função de cálculo
de juros compostos. O código gerado parece correto: a fórmula está certa, os
tipos estão corretos, o código compila sem erros. Ana, pressionada por um prazo
apertado, faz uma revisão rápida, pensa ``parece certo'', e integra o código ao
sistema. Três meses depois, a equipe de auditoria descobre que a função usa
aritmética de ponto flutuante para cálculos monetários, introduzindo erros de
arredondamento que, acumulados ao longo de milhares de transações, geraram uma
divergência de R\$ 47.000. Se Ana tivesse revisado o código linha por linha ---
ou, melhor ainda, pedido ao modelo ``use Decimal para todos os cálculos
monetários, nunca float'' ---, o problema teria sido evitado.

A disciplina de revisão da primeira resposta envolve várias práticas complementares.
Leia o código linha por linha, como faria em um code review de código humano. A
IA pode gerar código que compila mas tem bugs lógicos sutis, como condições de
borda não tratadas, off-by-one errors, race conditions em código concorrente.
Questione as decisões do modelo: ``por que você usou recursão aqui em vez de
iteração? Qual o comportamento com input vazio? E se a lista tiver um milhão de
elementos, a recursão estoura a stack?''. Peça alternativas: ``mostre outra
abordagem para este problema; quais são os trade-offs de cada uma?''. E, acima de
tudo, execute testes: código gerado por IA deve passar pelo mesmo rigor de
verificação que código humano. Se você não testaria código de um colega sem
rodá-lo, não deveria testar código de IA sem rodá-lo.

\subsection{Anti-padrão 4: Não Fornecer Exemplos}

Quando o formato de saída importa, descrever é inferior a demonstrar. Esse
anti-padrão é uma instância específica de um princípio mais geral: humanos (e
LLMs) aprendem melhor por exemplos do que por descrições abstratas.

Pedro precisa gerar documentação para as funções do módulo de pagamentos. Ele
digita: ``Gere documentação para esta função''. O modelo gera documentação no
estilo que considera ``padrão'', talvez JSDoc genérico, talvez Google style,
talvez uma mistura inconsistente. O resultado não se parece em nada com a
documentação que já existe no projeto de Pedro, e ele gasta tempo reformatando
cada bloco de documentação para manter a consistência.

A solução é trivial e poderosa: inclua um exemplo. ``Gere documentação para esta
função seguindo este formato: [cole um exemplo de documentação existente do
projeto]''. Com o exemplo, o modelo replica o estilo exato (cabeçalho,
estrutura de parâmetros, formato de exemplos, nível de detalhe) e o resultado
é imediatamente consistente com o restante do projeto. O custo de incluir um
exemplo é dezenas de tokens; a economia é dezenas de minutos de reformatação.

\subsection{Anti-padrão 5: Super-restrição vs. Sub-restrição}

Existe um equilíbrio delicado entre restringir demais e restringir de menos um
prompt. Sub-restrição produz resultados genéricos e inadequados, como já
discutimos nos anti-padrões anteriores. Mas super-restrição é igualmente
problemática: quando o prompt é tão detalhado e tão restritivo que não deixa
espaço para o modelo contribuir com insights, perde-se a principal vantagem de
usar IA, a capacidade de explorar soluções que o engenheiro não teria
considerado sozinho.

Imagine um prompt que especifica não apenas os requisitos, mas também a
implementação completa: ``use HashMap com chave String e valor Integer, itere com
for-each, use StringBuilder para concatenação, retorne o resultado como Optional
String''. Nesse ponto, o engenheiro já fez todo o trabalho de design; o modelo é
reduzido a um digitador sofisticado. Compare com um prompt que especifica os
requisitos e as restrições, mas deixa o design ao modelo: ``implemente uma função
que agrupe itens por categoria e retorne um resumo formatado; deve ser thread-safe,
não usar estado mutável compartilhado, e ter complexidade O(n)''. O segundo prompt
permite que o modelo escolha as estruturas de dados e os idiomas mais adequados,
potencialmente produzindo uma solução mais elegante ou mais eficiente do que a que
o engenheiro teria prescrito.

A regra prática é: restrinja os requisitos (o ``o quê'' e o ``quão bem''), mas
deixe flexibilidade no design (o ``como''). Especifique interfaces, contratos,
restrições de performance e padrões obrigatórios, mas não especifique a
implementação interna a menos que haja uma razão técnica concreta para fazê-lo.

\section{Conclusão}
% ============================================================

Prompt engineering é engenharia de comunicação. Assim como um engenheiro não
entrega uma especificação ambígua ao time de construção, não deveria entregar
um prompt ambíguo à IA.

Ao longo deste capítulo, exploramos um argumento central que merece ser
cristalizado: prompts são especificações. Não é uma metáfora. É uma
equivalência funcional. Um prompt bem escrito possui as mesmas qualidades de um
documento de requisitos bem escrito: é preciso, completo, não ambíguo e
verificável. Trate seus prompts com o mesmo rigor que daria a um documento de
requisitos. Revise-os. Peça a um colega que leia e diga se entendeu a mesma
coisa que você quis dizer. Teste-os contra critérios de aceitação definidos.
Versione-os. Essa mudança de mentalidade, de ``conversa casual com IA'' para
``especificação de engenharia'', é a transformação mais impactante que um
engenheiro pode fazer em sua prática de uso de LLMs.

A qualidade do prompt determina a qualidade do resultado. Essa correlação é
quase linear: prompts vagos produzem resultados vagos; prompts específicos
produzem resultados específicos; prompts com contexto rico produzem resultados
contextualizados. A IA reflete a precisão (ou a vagueza) da sua comunicação como
um espelho amplificador. Se sua comunicação é precisa, a amplificação trabalha a
seu favor; se é vaga, a amplificação trabalha contra você, produzindo respostas
que parecem detalhadas mas são fundamentalmente desconectadas do seu contexto.

A iteração é parte integrante do processo. O primeiro prompt raramente é o melhor,
assim como o primeiro rascunho de um código raramente é o definitivo. Refine,
teste, melhore. Aplique ao prompt o mesmo ciclo de desenvolvimento que aplica
ao código. Cada iteração revela algo sobre o problema que não era evidente antes,
e cada insight é incorporado no próximo prompt. Engenheiros que tratam prompts
como artefatos descartáveis perdem a oportunidade de construir uma base de
conhecimento cumulativa; engenheiros que versionam, documentam e refinam seus
prompts constroem uma biblioteca de templates que se torna cada vez mais valiosa
com o tempo.

Construir uma biblioteca de prompts é, portanto, uma prática de alto retorno.
Mantenha seus melhores prompts em um repositório, organizados por categoria
(revisão, geração, arquitetura, refatoração). Documente cada template com seu
contexto de uso, suas variações e seus resultados típicos. Compartilhe com a
equipe. Uma biblioteca de prompts compartilhada é para o uso de IA o que um
design system é para o desenvolvimento de interfaces: um investimento em
consistência e produtividade que se paga multiplicado ao longo do tempo.

O meta-skill que emerge de tudo isso é o saber \textbf{quando} e \textbf{como}
usar IA. Nem toda tarefa se beneficia de um prompt elaborado; nem toda decisão
deve ser delegada a um modelo. A habilidade de discernir quando a IA agrega valor
(exploração de alternativas, geração de boilerplate, revisão contra checklists) e
quando ela atrapalha (decisões que dependem de contexto organizacional, problemas
que exigem intuição humana desenvolvida ao longo de anos de experiência no
domínio) é o que separa o engenheiro que desperdiça tempo com IA do engenheiro que
multiplica sua produtividade com ela.

No próximo capítulo, aplicaremos esses padrões de prompting ao contexto
específico de revisão de código, onde a IA pode ser uma segunda opinião
valiosa, desde que seja instruída corretamente.

\section*{Exercícios}
% ============================================================

\begin{enumerate}
    \item \textbf{Anatomia de prompt}: Pegue um prompt que você usou
    recentemente com uma IA e reescreva-o usando o framework CRISP. Compare os
    resultados obtidos com a versão original e a versão CRISP. Documente as
    diferenças em qualidade de resposta.

    \item \textbf{Exploração arquitetural}: Escolha um sistema que você conhece
    bem (pode ser do trabalho ou um projeto pessoal). Escreva um prompt para
    pedir à IA que analise 3 alternativas arquiteturais para um componente
    específico. Avalie se a IA identificou alternativas que você não tinha
    considerado e se os trade-offs apontados são realistas.

    \item \textbf{Geração com testes}: Usando o template de geração de código
    com testes deste capítulo, peça à IA para implementar uma funcionalidade
    com testes. Execute os testes gerados. Quantos passam de primeira? Quais
    foram os problemas encontrados? Refine o prompt e repita.

    \item \textbf{Chain of Thought}: Pegue uma decisão técnica que sua equipe
    tomou recentemente. Reconstrua o processo usando um prompt com Chain of
    Thought. A IA chegou à mesma conclusão? Se não, os argumentos alternativos
    são válidos? Documente o processo em formato de ADR.

    \item \textbf{Anti-padrões na prática}: Colete 5 prompts ``ruins'' (vagos,
    sem contexto, genéricos) e reescreva cada um como prompt ``bom''. Para cada
    par, envie ambas as versões à IA e compare as respostas. Crie uma tabela
    documentando: prompt original, prompt melhorado, diferença qualitativa na
    resposta.

    \item \textbf{Biblioteca de prompts}: Crie uma biblioteca pessoal de
    prompts com pelo menos 10 templates reutilizáveis para suas tarefas mais
    frequentes (revisão de código, geração de testes, refatoração, documentação,
    análise de arquitetura). Para cada template, inclua: nome, descrição, prompt
    completo com placeholders, e exemplo de uso preenchido.
\end{enumerate}
